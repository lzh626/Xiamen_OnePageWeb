## 数据库设计

```
attractions (id, name, description, cover_image, region, rec_time, open_time, close_time,
    recommended_hours, suitable_weather, intensity_level, popularity_score, recommend_score,
    created_at, updated_at)
    ├──< attraction_tags >── tags (id, name, created_at, updated_at)
    ├──< comments (id, attraction_id, user_name, content, submission_token UNIQUE, ...)
    ├──< favorites (id, attraction_id, device_id, UNIQUE(attraction_id, device_id), ...)
    ├──< likes (id, attraction_id, device_id, UNIQUE(attraction_id, device_id), ...)
    ├──< route_items >── routes (id, name, duration_type, preferences, reason_text,
    │       weather_summary, ...)
    └──  weather_cache (city PK, payload TEXT, updated_at)
```

**设计理由：**
- `attraction_tags` 桥接表支持多对多标签，符合文档建议
- `likes` 独立于 `favorites`：点赞代表"推荐"，收藏代表"想去的"，语义不同，权重不同
- `routes` + `route_items` 拆分为主从表：支持按景点查询路线、修改路线顺序
- `weather_cache` 独立于景点：天气是城市级数据，与景点解耦

## 异常处理覆盖

| 异常场景 | 处理 | 现场演示方法 |
|:---|:---|:---|
| 天气接口超时 | 过期缓存 → fallback | 设 `WEATHER_TIMEOUT_SECONDS=1` |
| 温度占位值 999 | `_sanitize_temp` 过滤 → current_temp±4 推算 | 正常展示即可验证 |
| 温度高低倒挂 | 自动交换 | 正常展示即可验证 |
| 重复评论 | 409 Conflict | 连续提交两次同内容评论 |
| 重复收藏/点赞 | 幂等返回 duplicated=true | 连续点击两次收藏 |
| 非法区域参数 | 400 + "无效的区域参数，可选值:..." | 筛选栏输入不存在的区域 |
| 非法排序字段 | 400 | 直接调 `?sort_by=xxx` |
| 景点不存在 | 404 | 访问 `/api/attractions/99999` |
| 搜索无结果 | "没有找到匹配的景点" | 搜索"不存在的景点名" |
| LLM 失败 | 自动降级规则引擎 | 设 `RECOMMENDER_TYPE=llm` 但 `LLM_API_KEY` 为空 |
| 前端请求失败 | try-catch + 红色提示 | 关闭后端服务再刷新 |

## 面试关键问答预案

### Q1：为什么用 FastAPI 而不是 Flask/Django？
> FastAPI 原生支持 Pydantic 数据校验、自动 OpenAPI 文档、async/await 高性能。本项目接口以 JSON RESTful 为主，不需要 Django 的 ORM/Admin，Flask 需要手动拼装饰器。FastAPI 的 `Query()` 参数校验可以直接生成 422 错误 → 全局异常处理器捕获，代码量少一半。

### Q2：为什么点赞和收藏分开两个表？
> 语义不同：点赞代表"我推荐别人去"（权重×4），收藏代表"我以后想去"（权重×2）。分开后 popularity_score 可以按实际业务含义加权计算，而不是笼统计数。表结构也留了扩展空间——比如未来加入"浏览数""分享数"等更多行为维度。

### Q3：路线推荐的评分权重怎么确定的？
> 权重设计遵循"用户偏好 > 天气适配 > 基础热度"的优先级。天气适配 25 分（因为文档明确要求考虑天气影响），偏好匹配 15 分/项，基础热度 5-20 分。这是一个业务规则值，不是算法训练值——面试中重点看能否解释"为什么这样设计"，而不是追求数学完美。

### Q4：如何处理 LLM 失败？
> LLM 调用发生 RuntimeError（超时、密钥无效、返回非 JSON）时，`LLMRecommender.recommend()` 捕获异常后调用 `self._fallback = RuleRecommender()`，返回时标记 `"recommender": "rule-fallback"` 和 `"llm_error": true`。前端无感知，路线仍然可以展示。

### Q5：15 个景点的数据怎么来的？
> 都是真实厦门景点，按区域分布：思明区 9 个（鼓浪屿/厦大/环岛路/南普陀/曾厝垵/沙坡尾/中山路/胡里山/铁路公园/东坪山）、集美区 2 个（集美学村/园博苑）、湖里区 1 个（五缘湾）、海沧区 2 个（海沧大桥/海沧湾）。标签分配确保 6 种偏好每种 ≥3 个景点可用。`popularity_score` 基于种子点赞/收藏/评论数据按公式计算。

### Q6：前后端如何保证数据一致性？
> 统计数（comment_count/favorite_count/like_count）不由前端传递，而是后端 SQL `LEFT JOIN + COUNT GROUP BY` 实时聚合。点赞/收藏/评论写入成功后，`popularity_score` 在同一个事务中通过 UPDATE 子查询同步更新。前端只有显示层的即时更新（文本替换），真值始终以后端返回为准。

### Q7：为什么不直接使用大模型做路线推荐？
> 阶段 4 已实现 LLM 适配层，默认仍用规则引擎。理由：1. 规则版结果可解释、可测试、不受外部服务影响；2. 面试文档要求"可替换可降级"，当前架构恰好体现了这一原则——规则引擎作为兜底，LLM 作为增强；3. 现场演示时规则引擎 100% 可用，不依赖网络和 API Key。

### Q8：技术选型中的 AI 工具和提示策略
> 所有代码由 TRAE IDE 辅助生成。提示策略：分阶段迭代（一次只做一个需求块）→ 先写测试再写代码 → 异常优先（每次要求考虑失败场景）。人工校验：每个接口用 pytest 验证；前端用 F12 Network/Console 验证请求链路；天气用真实山河 API 返回数据校验字段；统计用终端 SQL 查询对比前端显示。

### Q9：AI 推荐的"自然语言出行需求理解"是必须的吗？
> **标准答案：**
> 不是必须的。面试文档原文写的是"**如果**使用大模型，需要做成可替换、可降级的模块"——说明这是架构能力要求，不是算法效果要求。你理解的"理解自然语言、给出生动答复"来自《实验7 综合实践》中「Agent 拓展加分项」（如 LangChain SQL Agent 做 NL2SQL 对话），那是可选拓展，不是面试硬性要求。当前项目实现的可替换适配层已经完整覆盖了文档第5条的三个核心点：1. 统一推荐接口可接多种实现；2. 可降级（LLM失败→规则引擎）；3. API Key 仅在后端环境变量。如果未来需要接入自然语言理解推荐，只需新增一个 `NLRecommender` 子类实现 `recommend()` 方法即可，主流程不改一行代码。

### Q10：.env 文件读取机制是什么？
> 项目早期存在设计缺陷——`settings.py` 使用 `os.getenv()` 只能读取系统环境变量，`.env.example` 只是模板从未被解析。修复：在 `config/settings.py` 第 4-7 行引入 `python-dotenv` 的 `load_dotenv(BASE_DIR / ".env")`，并在项目根目录创建真正的 `.env` 文件（已加入 `.gitignore`）。`requirements.txt` 新增 `python-dotenv==1.2.2`。现在只需要在 `.env` 中修改 `RECOMMENDER_TYPE=mock` / `RECOMMENDER_TYPE=llm` 即可一键切换推荐器，保存后 uvicorn `--reload` 自动生效。

### Q11：Mock 推荐器的时长 bug 怎么修的？
> 初始 `MockLLMRecommender` 的 `estimated_duration` 硬编码为 `"2.5小时"`（半日游）或 `"5.0小时"`（一日游），不依赖实际选中景点。修复：改为贪心选择逻辑（同 RuleRecommender 的时间约束），按实际选中景点的 `recommended_hours` 动态求和。同时 `LLMRecommender` 也做了同样修复。前端路线推荐展示从纯文本列表升级为图文卡片布局（左图右文 + 彩色标签 + 推荐理由 chips + 描述两行省略）。
